ToDesk Latency Test + Bandwidth Calculator

အဝေးထိန်းခလုတ်သည် ချောမွေ့သည်ဟု ခံစားရသည်ရှိမရှိ- ကွန်ရက်အသွားအပြန် latency (RTT) နှင့် ရရှိနိုင်သော bandwidth တို့ဖြစ်သည်။ အောက်ဖော်ပြပါ ကိရိယာ နှစ်ခုစလုံးသည် သင့်ဘရောက်ဆာတွင် ပြည်တွင်း၌ လုပ်ဆောင်သည် — ① RTT ကို တိုင်းတာပြီး အများသူငါ အဆုံးမှတ်များစွာသို့ တုန်လှုပ်စေကာ အဝေးထိန်းခလုတ် ချောမွေ့မှုကို အဆင့်သတ်မှတ်သည်။ ② bandwidth ယူနစ်များကို ပြောင်းပြီး ပေးထားသော bandwidth မောင်းနှင်နိုင်သည့် ကြည်လင်ပြတ်သားမှုကို ခန့်မှန်းပါ။ အခြေခံအချက်များသည် Microsoft ၏တရားဝင် Remote Desktop docs နှင့် ToDesk ၏ specs များမှလာပါသည် (အောက်ခြေရှိရင်းမြစ်များကိုကြည့်ပါ)။

ကိရိယာ 1- ကွန်ရက်အသွားအပြန် (Ping) စမ်းသပ်မှု

ခလုတ်ကို နှိပ်လိုက်သည်နှင့် သင့်ဘရောက်ဆာသည် အများသူငှာ အဆုံးမှတ်တစ်ခုစီသို့ တောင်းဆိုချက်အနည်းငယ် ပေးပို့ပြီး အသွားအပြန်ခရီးကို အကြိမ်များစွာ ပို့ပေးပါသည်။ ကျွန်ုပ်တို့သည် အဆုံးမှတ်တစ်ခုလျှင် အကောင်းဆုံး RTT ကို ယူသည် (အစစ်အမှန်ကွန်ရက် အသွားအပြန်ခရီးနှင့် အနီးစပ်ဆုံး) နှင့် အောက်ဖော်ပြပါ latency အဆင့်များကို အသုံးပြု၍ အဝေးထိန်းခလုတ်ချောမွေ့မှုကို အဆင့်သတ်မှတ်ပါသည်။

မှတ်ချက်- ဤအရာသည် ကွန်ရက်အသွားအပြန်ကို အများသူငှာ အဆုံးမှတ်များဆီသို့ တိုင်းတာသည် — အဝေးထိန်းခလုတ်၏ ကွန်ရက်ခြေထောက်အတွက် ပရောက်စီတစ်ခု။ Real remote-control latency သည် ဖမ်းယူမှု၊ ကုဒ်/ကုဒ်နှင့် ဖော်ပြချိန်တို့ကို ပေါင်းထည့်သည်။ ရလဒ်များသည် သင့်ကွန်ရက်နှင့် အဆုံးမှတ်အကွာအဝေးပေါ်တွင် မူတည်သည်။

ကိရိယာ 2- bandwidth converter + အရည်အသွေး ခန့်မှန်းချက်

ဘန်းဝဒ်တန်ဖိုးကို ထည့်ပါ၊ ယူနစ်တစ်ခုကို ရွေးပါ၊ ၎င်းသည် ဘုံယူနစ်များအဖြစ်သို့ ပြောင်းလဲကာ — Microsoft ၏ တရားဝင် အဝေးထိန်း ဒက်စ်တော့ အကြံပြုချက်များကို အသုံးပြုကာ — ဤ bandwidth သည် 30fps တွင် မောင်းနှင်နိုင်သည့် ကြည်လင်ပြတ်သားမှုအား အကြမ်းဖျင်း ခန့်မှန်းသည်။

အဝေးထိန်းချိန်ညှိမှု (အဆင့်များ) ကိုဖတ်နည်း

အပြန်အလှန်အကျိုးသက်ရောက်သော အဝေးထိန်းခလုတ်သည် လှိုင်းနှုန်းထက် latency တွင် ပို၍ အကဲဆတ်သည်။ အကောင်းမွန်ဆုံး အတွေ့အကြုံအတွက် RTT ≤20ms လိုချင်သော အပြန်အလှန်တုံ့ပြန်မှု/အကူအညီပေးသည့် အခြေအနေများကို Microsoft က မှတ်သားထားသည်။ အောက်ပါဇယားသည် ထိုကျောက်ဆူးကို လုပ်ငန်းအတွေ့အကြုံနှင့် ပေါင်းစပ်ထားသည် — သင့်အဆင့်ကို ရှာပါ။

အဆင့်အသွားအပြန် RTTအဝေးထိန်း အတွေ့အကြုံဘာလုပ်ရမလဲ
မြတ်သော< 50 msလက်ငင်းအနီး၊ အဝေးထိန်းခလုတ်သည် စက်တွင်းကို ခံစားရသည်။ကွန်ရက်ခြေထောက်တွင် အားနည်းသောလင့်ခ်မရှိပါ။ သင်သည်မြင့်မားသော resolution / frame rate ကိုတွန်းနိုင်သည်။
ကောင်းတယ်။50–100 msအကြံပြုထားသော အတိုင်းအတာအတွင်း၊ နေ့စဉ် ချောမွေ့သည်။ရံဖန်ရံခါ အနည်းငယ် လိမ်းကျံပြီး ဘာမှ မပြင်းထန်ပါ။
မျှတတယ်။100–150 msစာရိုက်ခြင်း/ဆွဲယူသည့်အခါ သိသာထင်ရှားသော နောက်ကျခြင်းအလင်းအလုပ်အတွက်အဆင်ပြေ; ပိုနီးသော အဆုံးမှတ်ကို ကူညီပေးသည်။
ပျင်းရိသည်။150–200 msသိသာစွာ ဂယက်ထကာ ခြေရာခံမှု ညံ့ဖျင်းသည်။ကြိုးတပ်ထားသော / ပိုနီးသော node သို့ပြောင်းပါ။ jitter နှင့် packet ဆုံးရှုံးမှုကို စစ်ဆေးပါ။
ဆင်းရဲတယ်။≥ 200 msအချိန်ကြာလာသည်နှင့်အမျှ အသုံးမဝင်လုနီးပါး ပင်ပန်းသည်။ကွန်ရက်ကို ဦးစွာပြင်ပါ- ကြိုးတပ်ထားသော၊ ပိုနီးသော relay သို့မဟုတ် အခပေး ဆုံမှတ်တစ်ခုကို ပြင်ပါ။

Bandwidth မည်မျှ (ပစ်မှတ်ကြည်လင်ပြတ်သားမှုဖြင့်)

အောက်ဖော်ပြပါဇယားသည် Microsoft ၏တရားဝင်တစ်ပုံတစ်ပုံပြတ်သားမှုလှိုင်းနှုန်းအကြံပြုချက် (30fps၊ <0.1% ဆုံးရှုံးမှု) ဖြစ်သည်။ ခေတ်မီအဝေးထိန်းခလုတ် (ToDesk နှင့် အခြားအရာများ) သည် H.264/AVC-444 နှင့် AV1 လိုက်လျောညီထွေရှိသော ဘစ်နှုန်းကိုအသုံးပြုသည်၊ ထို့ကြောင့် စစ်မှန်သောလိုအပ်ချက်များသည် မကြာခဏနည်းပါးသည် (Microsoft ၏ဥပမာ- ဝဘ်ရှာဖွေမှု 6 Mbps မှ 1 Mbps ခန့်သို့ကျဆင်းသွားသည်)။

Bandwidth ကို အကြံပြုထားသည်။ပစ်မှတ် resolution (@30fps)အမှုတွဲသုံးပါ။

Microsoft ၏တရားဝင်ပုံရိပ်ပြတ်သားမှု → ဘန်းဝဒ်ဇယားကို အခြေခံမျဉ်းအဖြစ် အသုံးပြုကာ 2026 ခုနှစ်အတွက် ပုံမှန်ကမ္ဘာ့ကိန်းဂဏန်းများကို ပေးဆောင်ရန် ခေတ်ပေါ် လိုက်လျောညီထွေသောကုဒ်ဒစ်များဖြင့် ပေါင်းစပ်ထားသော (H.264/AVC-444၊ AV1; ToDesk သည် 30% မှ 50% ပိုမိုကောင်းမွန်သော လှိုင်းနှုန်းများ ထိရောက်မှုရှိသည်ဟု ဆိုသည်)။

ခန့်မှန်းချက်၊ ပထမပါတီတိုင်းတာခြင်းမဟုတ်ပါ။ အခြေခံ = Microsoft ၏ တရားဝင် အဝေးထိန်း ဒက်စ်တော့ ဇယား (@30fps); 2026 ခန့်မှန်းချက်သည် ခေတ်မီ လိုက်လျောညီထွေရှိသော ကုဒ်နံပါတ်ဖြင့် အပိုထပ်လောင်းပြီး ပြင်းအား၏ အကြမ်းဖျဉ်း အတိုင်းအတာတစ်ခုသာဖြစ်သည် — တကယ့် bandwidth သည် စခရင်ပေါ်ရှိ ရွေ့လျားမှု၊ ပက်ကက်ဆုံးရှုံးမှုနှင့် codec တို့နှင့် ကွဲပြားသည်။

မေးလေ့ရှိသောမေးခွန်းများ

အဝေးထိန်းစနစ်အတွက် မည်သည့် latency ကို ပုံမှန်အဖြစ်ရေတွက်သနည်း။
လက်မ၏စည်းမျဉ်းအတိုင်း- 50ms အောက် RTT သည် ချက်ချင်းနီးပါးခံစားရသည်၊ 100ms အောက်သည် အကြံပြုထားသည့်အကွာအဝေးဖြစ်သည်၊ 100-150ms ကိုအသုံးပြု၍ရနိုင်သော်လည်း၊ နောက်ကျနေသည်ဟုခံစားရသည်၊ 150ms ကျော်သည် ရှင်းလင်းပြတ်သားနေပြီး 200ms သို့မဟုတ် ထို့ထက်ပိုပါက အသုံးမပြုနိုင်လုနီးပါးဖြစ်သည်။ Microsoft သည် အလွန်ကောင်းမွန်သော အပြန်အလှန်တုံ့ပြန်မှု/အကူအညီပေးသည့်အခြေအနေများတွင် RTT ≤20ms ကို အကောင်းဆုံးအတွေ့အကြုံရရှိရန် လိုလားကြောင်းလည်း မှတ်သားထားသည်။
ToDesk အဝေးထိန်းခလုတ်သည် လှိုင်းနှုန်းမည်မျှ လိုအပ်သနည်း။
Microsoft ၏တရားဝင်အဝေးထိန်းဒက်စ်တော့အကြံပြုချက် (30fps)- 720p အတွက် 3 Mbps ခန့်၊ 1080p အတွက် 5 Mbps နှင့် 4K အတွက် 15 Mbps ခန့်။ ခေတ်မီအဝေးထိန်းထိန်းချုပ်မှုတွင် လိုက်လျောညီထွေ ကုဒ်ပြောင်းခြင်းကို အသုံးပြုသောကြောင့် အမှန်တကယ် လိုအပ်ချက်များမှာ မကြာခဏ နည်းပါးပါသည်။
ဘယ်အရာက ပိုအရေးကြီးလဲ၊ latency သို့မဟုတ် bandwidth
အပြန်အလှန်အကျိုးသက်ရောက်သော အဝေးထိန်းခလုတ်အတွက်၊ latency သည် ပို၍အရေးကြီးသည်- လုံလောက်သော bandwidth ရှိသော်လည်း၊ မြင့်မားသော latency/jitter သည် နှောင့်နှေးနေသေးသည်။ Microsoft ၏ ဒိုင်းနမစ်ဘစ်နှုန်းသည် အရာခပ်သိမ်းကို ချောမွေ့စေရန် RTT မြင့်မားနေချိန်တွင် အရည်အသွေးကို အမှန်တကယ် ကျဆင်းစေသည်။
ဤ latency test သည် အဘယ်အရာကို အမှန်တကယ် တိုင်းတာသနည်း။
သင့်ဘရောက်ဆာသည် အများသူငှာ အဆုံးအဖြတ်များဆီသို့ ကွန်ရက်အသွားအပြန်ခရီးကို တိုင်းတာသည် — အဝေးထိန်းစနစ်၏ ကွန်ရက်ခြေထောက်အတွက် ပရောက်စီတစ်ခု။ Real remote-control latency သည် ဖမ်းယူမှု၊ ကုဒ်/ကုဒ်နှင့် ဖော်ပြချိန်တို့ကို ပေါင်းထည့်သည်။
ကျွန်ုပ်၏ bandwidth မြင့်မားနေချိန်တွင် အဘယ်ကြောင့် နောက်ကျနေသေးသနည်း။
များသောအားဖြင့် ၎င်းသည် latency၊ တုန်လှုပ်ခြင်း သို့မဟုတ် packet ဆုံးရှုံးမှု၊ bandwidth မဟုတ်ပါ။ အဝေးထိန်းခလုတ်သည် RTT နှင့် တည်ငြိမ်မှုအတွက် အထိခိုက်မခံပါ။ တုန်လှုပ်ခြင်းသည် မြင့်မားသောပျမ်းမျှထက် ပိုနာကျင်သည်။
ToDesk သည် အရည်အသွေးအမြင့်မားဆုံးရောက်ရှိနိုင်သည့်အရာများ။
၎င်း၏တရားဝင်ဆိုက်မှ 8K၊ 360 FPS နှင့် 4:4:4 စစ်မှန်သောအရောင်ဆုံးရှုံးမှုမရှိသော (AV1 codec / ZeroSync အင်ဂျင်) သည် မှန်ကန်သောအစီအစဉ်နှင့် လုံလောက်သော bandwidth ပေးထားသည့်အထိ တောင်းဆိုထားသည်။
1080p ဗီဒီယိုသည် အဝေးထိန်းခလုတ်ကို လှိုင်းနှုန်းမည်မျှယူသနည်း။
Microsoft ၏ဥပမာ- မျက်နှာပြင်တစ်ဝက် 30fps ဗီဒီယိုသည် မူရင်းအားဖြင့် 8.5–9.5 Mbps ခန့်ဖြစ်သော်လည်း H.264/AVC-444 နှင့် 2.5–2.8 Mbps ခန့်သာ — codec သည် ကြီးမားသောကွာခြားချက်တစ်ခုဖြစ်သည်။
Mbps နှင့် MB/s ကို မည်သို့ပြောင်းရမည်နည်း။
1 MB/s = 8 Mbps နှင့် 1 Mbps = 0.125 MB/s ။ ဤစာမျက်နှာရှိ bandwidth converter သည် ဘုံယူနစ်အားလုံးအတွက် ၎င်းကို အလိုအလျောက် ပြုလုပ်ပေးပါသည်။

လျင်မြန်သော လက်တွေ့ကမ္ဘာ စမ်းသပ်မှု

  • စာရိုက်ခြင်းနှင့် ဆွဲယူခြင်း windows နှေးကွေးသော်လည်း အသုံးပြုနိုင်သည်- RTT သည် များသောအားဖြင့် 100-150ms; ပိုနီးသော အဆုံးမှတ် သို့မဟုတ် ကြိုးတပ်ချိတ်ဆက်မှုတစ်ခုသည် ၎င်းကို 100ms အောက်သို့ ပြန်ဆွဲယူလေ့ရှိသည်။
  • ရုပ်ပုံသည် မှုန်ဝါးနေပြီး သင်ရွေ့လျားမှုရပ်တန့်ပြီးနောက်မှသာ ထက်မြက်လာသည်- အကန့်အသတ်ရှိသော ဘန်းဝဒ်သည် လိုက်လျောညီထွေဖြစ်စေသော အတိုင်းအတာကို အစပျိုးပေးသည်။ အရည်အသွေးအဆင့်များနှင့်ကိုက်ညီသော bandwidth converter ကိုစစ်ဆေးပြီး လိုအပ်ပါက ပစ်မှတ် resolution ကိုလျှော့ချပါ။
  • RTT သည် မမြင့်သော်လည်း ကြွလာ၍ သွားသည်- တုန်လှုပ်ခြင်းကို အာရုံစိုက်ပါ — တုန်လှုပ်ခြင်းသည် မြင့်မားသော ပျမ်းမျှထက် ပိုနာကျင်သည်။ Wi-Fi နှောင့်ယှက်မှုကို စစ်ဆေးပါ၊ 5GHz သို့ပြောင်းပါ သို့မဟုတ် ကြိုးတပ်ပါ။
  • နယ်စပ်ဖြတ်ကျော်အဝေးထိန်းစနစ်သည် အလွန်နှေးကွေးသည်- သာမန် အများသူငှာ လမ်းလွှဲလမ်းလွှဲများဖြစ်သောကြောင့် RTT နှင့် ပက်ကက်ပျောက်ဆုံးမှု နှစ်ခုစလုံးသည် မြင့်မားသည်။ ပိုနီးသော relay သို့မဟုတ် ထုတ်ကုန်၏ ကမ္ဘာလုံးဆိုင်ရာ ဆုံမှတ်များက များစွာ အထောက်အကူ ပြုပါသည်။
  • 4K အဝေးထိန်းထိန်းချုပ်မှုတွင် လုံလောက်သော bandwidth ပါသော်လည်း frames များကို ကျဆင်းစေသည်- မြင့်မားသော res/frame rate သည် bandwidth သာမကဘဲ encode/decode compute ကို အခွန်ပေးဆောင်ပါသည်။ 1080p သို့မဟုတ် အောက်ဖရိမ်နှုန်းကို စမ်းကြည့်ပါ။

  • ToDesk တရားဝင်ဆိုက် (ထုတ်ကုန်သတ်မှတ်ချက်များ)
Microsoft ၏တရားဝင်အဝေးထိန်း ဒက်စ်တော့ဒေါ့စ် (ကွန်ရက်လမ်းညွှန်ချက်များ / Azure Virtual Desktop RDP ၏ လှိုင်းနှုန်း) မှ Bandwidth/resolution အခြေခံများ။ ၎င်း၏တရားဝင်ဆိုက်မှ ToDesk သတ်မှတ်ချက်များ။ Latency အဆင့်များသည် Microsoft ၏ RTT≤20ms anchor ဖြင့် စစ်ဆေးထားသော လုပ်ငန်းအတွေ့အကြုံ တီးဝိုင်းတစ်ခုဖြစ်သည်။ ဤနေရာတွင် ကိရိယာများနှင့် ခန့်မှန်းချက်များသည် ကိုးကားရန်အတွက်သာဖြစ်ပြီး ပထမပါတီတိုင်းတာခြင်းမဟုတ်ပါ။
