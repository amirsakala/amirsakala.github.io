ToDesk Latency Test + ເຄື່ອງຄິດເລກແບນວິດ

ບໍ່ວ່າການຄວບຄຸມທາງໄກຈະຮູ້ສຶກລຽບງ່າຍຢູ່ສອງຢ່າງຄື: ການຕອບສະໜອງການໄປມາຂອງເຄືອຂ່າຍ (RTT) ແລະ ແບນວິດທີ່ສາມາດໃຊ້ໄດ້. ເຄື່ອງ​ມື​ທັງ​ສອງ​ຂ້າງ​ລຸ່ມ​ນີ້​ດໍາ​ເນີນ​ການ​ໃນ​ທ້ອງ​ຖິ່ນ​ໃນ​ຕົວ​ທ່ອງ​ເວັບ​ຂອງ​ທ່ານ — ①​ການ​ວັດ​ແທກ RTT ແລະ jitter ກັບ​ຈຸດ​ສຸດ​ສາ​ທາ​ລະ​ນະ​ຈໍາ​ນວນ​ຫນຶ່ງ​ແລະ​ຈັດ​ອັນ​ດັບ​ຄວາມ​ສະ​ດວກ​ການ​ຄວບ​ຄຸມ​ຫ່າງ​ໄກ​ສອກ​ຫຼີກ​; ②​ແປງ​ຫົວ​ຫນ່ວຍ​ແບນ​ວິດ​ແລະ​ຄາດ​ຄະ​ເນ​ຄວາມ​ລະ​ອຽດ​ຂອງ​ແບນ​ວິດ​ທີ່​ໄດ້​ຮັບ​ການ​ຂັບ​ໄດ້​. ພື້ນຖານມາຈາກເອກະສານ Remote Desktop ຢ່າງເປັນທາງການຂອງ Microsoft ແລະຂໍ້ມູນຈໍາເພາະຂອງ ToDesk (ເບິ່ງແຫຼ່ງຂໍ້ມູນຢູ່ດ້ານລຸ່ມ).

ເຄື່ອງ​ມື 1​: ການ​ທົດ​ສອບ​ການ​ໄປ​ກັບ​ເຄືອ​ຂ່າຍ (Ping​)​

ກົດປຸ່ມແລະຕົວທ່ອງເວັບຂອງທ່ານຈະສົ່ງຄໍາຮ້ອງຂໍຈໍານວນຫນ້ອຍໄປຫາແຕ່ລະຈຸດສິ້ນສຸດສາທາລະນະແລະເວລາຂອງການເດີນທາງ. ພວກເຮົາເອົາ RTT ທີ່ດີທີ່ສຸດ ຕໍ່ຈຸດສິ້ນສຸດ (ໃກ້ທີ່ສຸດກັບການເດີນທາງໄປມາຂອງເຄືອຂ່າຍທີ່ແທ້ຈິງ) ແລະໃຫ້ຄະແນນຄວາມລຽບງ່າຍຂອງການຄວບຄຸມໄລຍະໄກໂດຍໃຊ້ລະດັບການຕອບສະໜອງຂ້າງລຸ່ມນີ້.

ໝາຍເຫດ: ອັນນີ້ວັດແທກ ການເດີນທາງໄປມາຂອງເຄືອຂ່າຍ ໄປຫາຈຸດສິ້ນສຸດສາທາລະນະ — ຕົວແທນສຳລັບຂາເຄືອຂ່າຍຂອງການຄວບຄຸມທາງໄກ. latency ການຄວບຄຸມໄລຍະໄກທີ່ແທ້ຈິງຍັງເພີ່ມການຈັບພາບ, ການເຂົ້າລະຫັດ / ຖອດລະຫັດແລະເວລາສະແດງ. ຜົນໄດ້ຮັບແມ່ນຂຶ້ນກັບເຄືອຂ່າຍ ແລະໄລຍະທາງປາຍທາງຂອງທ່ານ.

ເຄື່ອງມື 2: ຕົວແປງແບນວິດ + ການຄາດຄະເນຄຸນນະພາບ

ໃສ່ຄ່າແບນວິດ, ເລືອກຫົວໜ່ວຍໃດໜຶ່ງ, ແລະມັນປ່ຽນເປັນຫົວໜ່ວຍທົ່ວໄປ ແລະ — ໂດຍການນຳໃຊ້ຄຳແນະນຳທາງໄກຂອງເດັສທັອບທີ່ເປັນທາງການຂອງ Microsoft — ຄາດຄະເນວ່າຄວາມລະອຽດຂອງແບນວິດນີ້ສາມາດຂັບໄດ້ຢູ່ທີ່ 30fps.

ວິທີການອ່ານ latency ຄວບຄຸມໄລຍະໄກ (tiers)

ການຄວບຄຸມໄລຍະໄກແບບໂຕ້ຕອບແມ່ນມີຄວາມອ່ອນໄຫວຕໍ່ກັບເວລາ latency ຫຼາຍກ່ວາແບນວິດ. Microsoft ສັງເກດວ່າສະຖານະການທີ່ມີການໂຕ້ຕອບ / ການຊ່ວຍເຫຼືອສູງຕ້ອງການ RTT ≤20ms ສໍາລັບປະສົບການທີ່ດີທີ່ສຸດ. ຕາຕະລາງຂ້າງລຸ່ມນີ້ລວມເອົາສະມໍນັ້ນກັບປະສົບການອຸດສາຫະກໍາ - ຊອກຫາລະດັບຂອງທ່ານ.

ຊັ້ນໄປກັບ RTTປະສົບການການຄວບຄຸມໄລຍະໄກຈະເຮັດແນວໃດ
ເລີດ< 50 msການ​ຄວບ​ຄຸມ​ຫ່າງ​ໄກ​ສອກ​ຫຼີກ​ຢູ່​ໃກ້​ກັບ​ທັນ​ທີ​, ຮູ້​ສຶກ​ວ່າ​ຢູ່​ໃນ​ທ້ອງ​ຖິ່ນ​ບໍ່ມີການເຊື່ອມຕໍ່ທີ່ອ່ອນແອຢູ່ໃນຂາເຄືອຂ່າຍ; ທ່ານ​ສາ​ມາດ​ຊຸກ​ດັນ​ໃຫ້​ຄວາມ​ລະ​ອຽດ​ສູງ / ອັດ​ຕາ​ເຟຣມ​.
ດີ50–100 msພາຍໃນຂອບເຂດທີ່ແນະນໍາ, ກ້ຽງມື້ຕໍ່ມື້ເປັນບາງໂອກາດ smearing ເລັກນ້ອຍ, ບໍ່ມີຫຍັງຮ້າຍແຮງ.
ຍຸດຕິທຳ100–150 msຄວາມຊັກຊ້າທີ່ສັງເກດເຫັນໃນເວລາພິມ/ລາກການປັບໄຫມສໍາລັບວຽກງານແສງສະຫວ່າງ; ຈຸດສິ້ນສຸດທີ່ໃກ້ກວ່າຈະຊ່ວຍໄດ້.
ຂີ້ຄ້ານ150–200 msຢ່າງຈະແຈ້ງ, ຕິດຕາມບໍ່ດີສະຫຼັບໄປໃຊ້ສາຍ / a nearer node; ກວດເບິ່ງ jitter ແລະການສູນເສຍແພັກເກັດ.
ທຸກຍາກ≥ 200 msເກືອບໃຊ້ບໍ່ໄດ້, ເມື່ອຍຕາມເວລາແກ້ໄຂເຄືອຂ່າຍກ່ອນ: ມີສາຍ, relay ທີ່ໃກ້ກວ່າ, ຫຼື node ຈ່າຍ.

ແບນວິດເທົ່າໃດ (ຕາມຄວາມລະອຽດເປົ້າໝາຍ)

ຕາຕະລາງຂ້າງລຸ່ມນີ້ແມ່ນການແນະນໍາແບນວິດຕໍ່ການແກ້ໄຂຢ່າງເປັນທາງການຂອງ Microsoft (30fps, <0.1% ການສູນເສຍ). ການຄວບຄຸມໄລຍະໄກທີ່ທັນສະໄຫມ (ToDesk ແລະອື່ນໆ) ໃຊ້ອັດຕາບິດແບບປັບຕົວ H.264/AVC-444 ແລະ AV1, ດັ່ງນັ້ນຄວາມຕ້ອງການທີ່ແທ້ຈິງມັກຈະຕ່ໍາ (ຕົວຢ່າງຂອງ Microsoft: ການທ່ອງເວັບຫຼຸດລົງຈາກ 6 Mbps ໄປປະມານ 1 Mbps).

ການນໍາໃຊ້ຄວາມລະອຽດທີ່ເປັນທາງການຂອງ Microsoft →ຕາຕະລາງແບນວິດເປັນພື້ນຖານ, ເພີ່ມເຕີມດ້ວຍຕົວແປງສັນຍານການປັບຕົວທີ່ທັນສະໄຫມ (H.264/AVC-444, AV1; ToDesk ອ້າງວ່າປະສິດທິພາບແບນວິດທີ່ດີຂຶ້ນ 30%–50%) ເພື່ອໃຫ້ຕົວເລກທີ່ແທ້ຈິງທົ່ວໄປສໍາລັບປີ 2026.

ຄຳຖາມທີ່ມັກຖາມເລື້ອຍໆ

ເວລາແຝງໃດທີ່ນັບເປັນປົກກະຕິສໍາລັບການຄວບຄຸມທາງໄກ?
ຕາມກົດລະບຽບ: RTT ຕ່ໍາກວ່າ 50ms ຮູ້ສຶກວ່າຢູ່ໃກ້ກັບທັນທີ, ພາຍໃຕ້ 100ms ແມ່ນໄລຍະທີ່ແນະນໍາ, 100-150ms ແມ່ນສາມາດໃຊ້ໄດ້ແຕ່ທ່ານຮູ້ສຶກວ່າຄວາມຊັກຊ້າ, ຫຼາຍກວ່າ 150ms ແມ່ນຊ້າຢ່າງຊັດເຈນ, ແລະ 200ms ຫຼືຫຼາຍກວ່ານັ້ນເກືອບໃຊ້ບໍ່ໄດ້. Microsoft ຍັງສັງເກດວ່າສະຖານະການໂຕ້ຕອບ / ການຊ່ວຍເຫຼືອສູງຕ້ອງການ RTT ≤20ms ສໍາລັບປະສົບການທີ່ດີທີ່ສຸດ.
ການຄວບຄຸມໄລຍະໄກ ToDesk ຕ້ອງການແບນວິດຫຼາຍປານໃດ?
ຕາມຄໍາແນະນໍາຢ່າງເປັນທາງການຂອງ Microsoft Remote Desktop (30fps): ປະມານ 3 Mbps ສໍາລັບ 720p, 5 Mbps ສໍາລັບ 1080p ແລະ 15 Mbps ສໍາລັບ 4K. ການຄວບຄຸມໄລຍະໄກທີ່ທັນສະໄຫມໃຊ້ການເຂົ້າລະຫັດແບບປັບຕົວ, ດັ່ງນັ້ນຄວາມຕ້ອງການທີ່ແທ້ຈິງມັກຈະຕ່ໍາ.
ອັນໃດສຳຄັນກວ່າ, ຄວາມແຝງ ຫຼືແບນວິດ?
ສໍາລັບການຄວບຄຸມໄລຍະໄກແບບໂຕ້ຕອບ, ເວລາ latency ມີຄວາມສໍາຄັນຫຼາຍ: ເຖິງແມ່ນວ່າມີແບນວິດພຽງພໍ, latency / jitter ສູງຍັງມີຄວາມຮູ້ສຶກຊ້າ. ອັດຕາບິດແບບເຄື່ອນໄຫວຂອງ Microsoft ຕົວຈິງເຮັດໃຫ້ຄຸນນະພາບຕໍ່າລົງເມື່ອ RTT ສູງເພື່ອຮັກສາສິ່ງທີ່ລຽບງ່າຍ.
ການທົດສອບ latency ນີ້ວັດແທກອັນໃດແທ້?
ບຼາວເຊີຂອງທ່ານວັດແທກການເດີນທາງໄປມາຂອງເຄືອຂ່າຍໄປຫາຈຸດສິ້ນສຸດສາທາລະນະ (Cloudflare, Google, ແລະອື່ນໆ) — ຕົວແທນຂອງເຄືອຂ່າຍຄວບຄຸມໄລຍະໄກ. latency ການຄວບຄຸມໄລຍະໄກທີ່ແທ້ຈິງຍັງເພີ່ມການຈັບພາບ, ການເຂົ້າລະຫັດ / ຖອດລະຫັດແລະເວລາສະແດງ.
ເປັນຫຍັງມັນຍັງຊັກຊ້າເມື່ອແບນວິດຂອງຂ້ອຍສູງ?
ປົກກະຕິແລ້ວມັນເປັນການ latency, jitter ຫຼືການສູນເສຍແພັກເກັດ, ບໍ່ແມ່ນແບນວິດ. ການຄວບຄຸມໄລຍະໄກແມ່ນມີຄວາມອ່ອນໄຫວຕໍ່ກັບ RTT ແລະຄວາມຫມັ້ນຄົງ; jitter ສູງເຈັບປວດຫຼາຍກ່ວາສະເລ່ຍສູງ.
ToDesk ຄຸນນະພາບສູງສຸດແມ່ນຫຍັງທີ່ສາມາດບັນລຸໄດ້?
ເວັບໄຊທ໌ທາງການຂອງມັນອ້າງເຖິງ 8K, 360 FPS ແລະ 4: 4: 4 true-color lossless (AV1 codec / ZeroSync engine), ໂດຍໃຫ້ແຜນການທີ່ຖືກຕ້ອງແລະແບນວິດພຽງພໍ.
ວິ​ດີ​ໂອ 1080p ແບນ​ວິດ​ຫຼາຍ​ປານ​ໃດ​ເອົາ​ການ​ຄວບ​ຄຸມ​ໄລ​ຍະ​ໄກ​?
ຕົວຢ່າງຂອງ Microsoft: ວິດີໂອ 30fps ເຄິ່ງໜ້າຈໍແມ່ນປະມານ 8.5–9.5 Mbps ຕາມຄ່າເລີ່ມຕົ້ນ, ແຕ່ມີພຽງປະມານ 2.5–2.8 Mbps ກັບ H.264/AVC-444 — ຕົວແປງສັນຍານສ້າງຄວາມແຕກຕ່າງອັນໃຫຍ່ຫຼວງ.
ຂ້ອຍຈະແປງ Mbps ແລະ MB/s ໄດ້ແນວໃດ?
1 MB/s = 8 Mbps, ແລະ 1 Mbps = 0.125 MB/s. ຕົວແປງແບນວິດໃນຫນ້ານີ້ເຮັດມັນສໍາລັບຫນ່ວຍງານທົ່ວໄປທັງຫມົດໂດຍອັດຕະໂນມັດ.

triage ໂລກ​ທີ່​ແທ້​ຈິງ​ໄວ​

  • ການພິມ ແລະລາກ windows ຊ້າແຕ່ສາມາດໃຊ້ໄດ້: RTT ປົກກະຕິແມ່ນ 100–150ms; ຈຸດສິ້ນສຸດທີ່ໃກ້ກວ່າ ຫຼືການເຊື່ອມຕໍ່ແບບມີສາຍມັກຈະດຶງມັນກັບຄືນພາຍໃຕ້ 100ms.
  • ຮູບພາບແມ່ນມົວ ແລະພຽງແຕ່ແຫຼມ ຫຼັງຈາກທີ່ທ່ານຢຸດການເຄື່ອນຍ້າຍ: ແບນວິດທີ່ຈໍາກັດເຮັດໃຫ້ການປັບຂະຫນາດຫຼຸດລົງ; ກວດເບິ່ງຕົວແປງແບນວິດທຽບກັບລະດັບຄຸນນະພາບ ແລະຫຼຸດຄວາມລະອຽດເປົ້າໝາຍຖ້າຕ້ອງການ.
  • RTT ບໍ່ສູງແຕ່ມັນມາແລະໄປ: ສຸມໃສ່ການ jitter - jitter ສູງເຈັບປວດຫຼາຍກ່ວາສະເລ່ຍສູງ; ກວດເບິ່ງການລົບກວນ Wi-Fi, ປ່ຽນເປັນ 5GHz ຫຼືມີສາຍ.
  • ການຄວບຄຸມຫ່າງໄກສອກຫຼີກຂ້າມຊາຍແດນແມ່ນ laggy ຫຼາຍ: ເສັ້ນທາງຍ່າງທາງສາທາລະນະທໍາມະດາ, ດັ່ງນັ້ນ RTT ແລະການສູນເສຍແພັກເກັດແມ່ນສູງ; relay ທີ່ໃກ້ກວ່າຫຼື nodes ທົ່ວໂລກຂອງຜະລິດຕະພັນຊ່ວຍຫຼາຍ.
  • ການຄວບຄຸມໄລຍະໄກ 4K ມີແບນວິດພຽງພໍແຕ່ຫຼຸດລົງເຟຣມ: ອັດຕາ res / ເຟຣມສູງຍັງພາສີ encode / decode compute, ບໍ່ພຽງແຕ່ແບນວິດ; ລອງໃຊ້ 1080p ຫຼືອັດຕາເຟຣມຕໍ່າກວ່າ.

  • ເວັບໄຊຢ່າງເປັນທາງການຂອງ ToDesk (ລາຍລະອຽດຂອງຜະລິດຕະພັນ)
ແບນວິດ / ຄວາມລະອຽດພື້ນຖານຈາກເອກະສານ Remote Desktop ຢ່າງເປັນທາງການຂອງ Microsoft (ຂໍ້ແນະນໍາເຄືອຂ່າຍ / Azure Virtual Desktop RDP bandwidth); ToDesk specs ຈາກເວັບໄຊຢ່າງເປັນທາງການຂອງຕົນ. Latency tiers ແມ່ນແຖບປະສົບການຂອງອຸດສາຫະກໍາທີ່ກວດສອບຜ່ານ RTT≤20ms ຂອງ Microsoft. ເຄື່ອງມືແລະການຄາດຄະເນນີ້ແມ່ນສໍາລັບການອ້າງອີງເທົ່ານັ້ນ, ບໍ່ແມ່ນການວັດແທກຂອງບຸກຄົນທໍາອິດ.
